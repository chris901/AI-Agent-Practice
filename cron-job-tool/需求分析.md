# 对话历史与知识检索（RAG）功能需求分析

## 1. 文档说明


| 项目   | 说明                                            |
| ---- | --------------------------------------------- |
| 文档目的 | 明确「历史记录」与「RAG 检索」在业务与前端侧应满足的能力与体验，作为产品评审与迭代依据 |
| 适用范围 | 智能助手类对话场景（含多轮对话、工具调用、定时任务等能力）                 |
| 不包含  | 具体技术架构、接口字段、数据库设计、选型与实现方案                     |


---

## 2. 背景与目标

### 2.1 背景

用户与助手进行多轮交互时，存在两类典型痛点：

- **上下文断裂**：刷新页面、换设备或隔日再来时，无法延续此前约定与结论。
- **长对话中「记不清前文」**：即使用户记得自己说过什么，模型在单次请求中未必能稳定利用全部历史，需要一种「按语义找回相关片段」的能力，而不是仅靠把整段历史原样堆进输入框。

### 2.2 目标

- **历史记录**：让用户能查看、区分、继续某次对话，并在产品层形成可追溯的对话脉络。
- **RAG 检索**：在对话过程中，系统能基于「与当前问题语义相近」的历史内容辅助回答，使用户感受到「助手记得我们聊过什么」，且回答有据可依、减少胡编。

---

## 3. 名词说明（产品语义）


| 名词     | 产品含义                                                                 |
| ------ | -------------------------------------------------------------------- |
| 会话     | 用户与助手之间的一条连续对话线，有明确开始，可包含多轮问答                                        |
| 历史记录   | 以会话为维度保存的用户消息与助手回复，支持按时间顺序查看                                         |
| RAG 检索 | 根据用户当前输入，从历史（通常优先本会话）中取出与当前问题最相关的若干片段，供助手生成回答时参考（用户侧可感知为「结合历史」的智能回复） |


---

## 4. 功能需求：历史记录

### 4.1 会话管理

- 用户应能**新建对话**，每次新建对应一个新的会话，与旧会话在列表上可区分。
- 用户应能查看**会话列表**（至少包含：会话标识、最近一条预览或标题、最近活跃时间），便于快速找到某次对话。
- 用户应能**进入某一历史会话**，继续在该会话下提问；继续对话时，新内容应归属该会话，而不是误写入其他会话。
- 用户应能**区分「当前正在聊的会话」**与「仅浏览的历史会话」（避免误发消息到错误会话）。

### 4.2 消息展示

- 在单个会话内，消息应按**时间顺序**展示，并明确区分**用户消息**与**助手消息**。
- 单条消息过长时，前端应支持**阅读体验**上的处理（如折叠、展开），具体展示策略由交互设计确定，需求上要求「可读、可定位」。
- 若业务中存在「工具执行结果」「系统提示」等对用户可见的补充信息，应在需求中约定是否展示、如何与助手正文区分（本阶段若未上线，可作为后续迭代项列出）。

### 4.3 会话生命周期（产品规则）

- 需明确：**删除会话**是否提供；若提供，删除后列表与详情是否立即一致，以及是否给予二次确认。
- 需明确：**清空当前会话上下文**（若有）与「删除会话」是否为不同操作，避免用户混淆。

### 4.4 与「继续对话」的关系

- 用户从会话列表点进某条历史会话后，发送的新问题必须视为**该会话的下一轮**，历史列表中应能看到新增的用户消息与助手回复。
- 新建会话后，**不应**默认混入其他会话的历史内容（除非产品单独定义「跨会话引用」能力）。

---

## 5. 功能需求：RAG 检索

### 5.1 检索范围（产品策略）

- 需约定：RAG 默认**优先使用当前会话内的历史**，还是允许包含**其他会话**中的内容。不同选择对应不同的隐私预期与产品文案（例如是否提示「已参考其他对话」）。
- 若存在「仅本会话」与「全局知识」两种模式，产品需定义：**默认模式**、**用户是否可切换**、切换入口放在哪里。

### 5.2 用户可感知行为

- 当助手回答明显依赖历史中的事实（例如用户曾提供的姓名、偏好、截止时间）时，用户应感受到**一致性**与**记忆感**。
- 产品可（可选）提供**弱提示**：例如「已参考本对话中的相关内容」，用于建立信任；是否展示、文案强度由产品决定，需求上需「不误导」（未检索到内容时不应假装已参考）。

### 5.3 边界与体验

- 当历史很少或首次提问时，助手行为应与「无额外检索」时一致，**不应**出现无意义的重复提示或错误提示。
- 当用户问题与历史无关时，助手应正常回答，**不应**强行拼凑历史片段导致答非所问。

### 5.4 与历史记录的关系

- RAG 是对**已写入历史的语义化利用**；用户在前端看到的「历史消息列表」仍是权威展示，RAG 是后台对片段的选取与利用，**不要求**逐条向用户展示「被检索到的片段」（除非产品明确要求「可解释性」视图）。

## 6. 前端使用场景与交互需求（怎么用）

以下从**用户任务**出发描述前端应具备的能力，不涉及实现方式。

### 6.1 首次进入 / 无会话

- 用户进入对话页时，应有明确状态：**当前无会话**或**自动进入新会话**，并在界面上让用户知道「现在是一条新对话」。
- 首次发送问题后，该产品应形成**第一条会话记录**，并在侧边栏或列表中可见（若采用列表形态）。

### 6.2 多轮连续对话

- 用户在同一页面连续多轮输入时，界面应持续追加消息，且**会话维度一致**（即属于同一会话）。
- 用户切换浏览器标签或短暂离开后返回，若产品承诺「恢复现场」，则应能恢复**同一会话与消息列表**（具体是否持久化到服务端由非本文件范围约定，此处为体验目标）。

### 6.3 从历史继续聊

- 用户从「历史会话列表」选中某会话后，主区域应加载该会话的**完整消息历史**（或分页加载，但需求上应保证可追溯）。
- 输入框发送的内容应对应该会话；列表中应高亮或标记**当前会话**，避免串会话。

### 6.4 新建与旧会话并存

- 用户点击「新对话」时，应**新开会话**，主区域清空或进入空白新会话，旧会话仍保留在列表中且可再次进入。
- 需求上应明确：**是否允许并行多个「未结束的草稿会话」**，以及列表排序规则（例如按最近活跃时间）。

### 6.5 流式输出场景

- 若助手回复为流式展示，历史记录应在**本轮回复结束后**与列表一致：用户应看到完整助手消息出现在时间线中，而不是长期停留在半成品状态（失败场景需单独约定，见下节）。

### 6.6 异常与加载

- 加载历史列表、加载某会话消息时，应有**加载中**状态；失败时应有**可重试或明确错误说明**。
- 发送失败时，应明确是「未写入历史」还是「已保存用户消息但助手失败」，避免用户以为助手已答复。

---

## 7. 验收关注点（产品侧）

- 会话列表与会话内消息是否**一一对应、顺序正确**。
- 切换会话后发送消息，是否**始终落在当前选中的会话**。
- 在长对话、隔多轮追问后，助手是否在**合理场景**下体现出对前文的「语义记忆」（可通过典型用例脚本验收）。
- RAG 相关：在「仅本会话」策略下，**不应**出现明显引用其他会话隐私内容的情况（若未做跨会话检索）。

---

## 8. 待产品补充的决策点

以下事项影响前端与文案，需在立项或迭代时拍板：

1. 会话是否需要**重命名 / 自动生成标题**（便于列表辨认）。
2. RAG 是否对用户展示「参考了历史」的提示，以及提示文案。
3. 是否提供**删除单条消息**、**编辑用户消息**（若提供，与历史及 RAG 一致性需另立规则）。
4. 移动端与桌面端是否同一套会话管理形态，或简化形态。
